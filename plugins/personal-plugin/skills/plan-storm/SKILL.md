# Plan Storm

N명의 에이전트가 독립적으로 구현 계획을 수립하고, 교차 리뷰를 통해 검증된 최종 계획서를 만드는 워크플로우.

각 에이전트는 Architecture, Implementation, Review 3가지 관점을 모두 포함하여 분석한다. 전문 역할이 나뉘는 것이 아니라, 모든 에이전트가 동일한 범위를 독립적으로 커버함으로써 누락과 편향을 줄인다.

---

## Step 0: 설정 입력 및 확인

스킬을 실행하기 전에 **반드시** 사용자에게 설정을 입력받고 확인을 받는다.

AskUserQuestion 도구를 사용하여 다음 항목을 입력받는다:

### 0-1. 에이전트 수 입력

```
Plan Storm 에이전트 수를 설정합니다.
- 동시에 실행할 에이전트 수를 입력하세요. (기본값: 3, 범위: 2~5)
```

- 사용자가 값을 입력하지 않거나 빈 값이면 기본값 **3**을 사용한다.
- 입력값이 2 미만이면 2로, 5 초과면 5로 조정하고 사용자에게 알린다.
- 이후 이 값을 `{N}`으로 참조한다.

### 0-2. 모델 선택

```
사용할 모델을 선택하세요:

[Claude 모델]
  1. opus   — 가장 정밀한 분석 (기본값)
  2. sonnet — 균형 잡힌 성능
  3. haiku  — 빠른 응답 (가벼운 작업용)

[LM Studio]
  4. lmstudio — LM Studio에 현재 로드된 모델 사용 (localhost:1234)

번호 또는 모델명을 입력하세요. (기본값: 1/opus)
```

- 사용자가 값을 입력하지 않거나 빈 값이면 기본값 **opus**를 사용한다.
- `lmstudio` 선택 시, 먼저 LM Studio API 연결을 확인한다 (Step 0-3 참조).
- 이후 이 값을 `{MODEL}`로 참조한다.

### 0-3. LM Studio 연결 확인 (lmstudio 선택 시에만)

`{MODEL}`이 `lmstudio`인 경우, Bash 도구로 다음을 실행하여 연결 및 로드된 모델을 확인한다:

```bash
curl -s --max-time 10 http://localhost:1234/v1/models
```

- 연결 확인 타임아웃은 10초로 설정한다.
- 응답이 없거나 오류가 발생하면 사용자에게 알리고 모델을 다시 선택하도록 한다.
- 정상 응답이면 로드된 모델 목록을 보여주고, 현재 로드된 모델명을 `{LMSTUDIO_MODEL}`에 저장한다.

### 0-4. 최종 확인

```
Plan Storm을 실행합니다.
- 입력: {이슈 URL 또는 작업 설명 요약}
- 에이전트 수: {N}명
- 모델: {MODEL} {lmstudio인 경우: "({LMSTUDIO_MODEL})"}
- {N}명의 에이전트가 병렬로 계획을 수립하고 교차 리뷰합니다.
- 최종 계획서가 .claude/tasks/{title}.md에 저장됩니다.

진행할까요?
```

사용자가 승인하면 다음 단계로 진행한다. 거부하면 즉시 중단한다.

---

## Step 1: 입력 수집

사용자로부터 다음 중 하나를 받는다:

- **GitHub 이슈 URL**: `gh issue view` 또는 `WebFetch`로 이슈 내용을 가져온다
- **작업 설명 텍스트**: 사용자가 직접 작성한 요구사항

입력이 불충분하면 추가 정보를 요청한다. 이슈 URL이 주어진 경우, 이슈 본문뿐 아니라 댓글도 확인하여 맥락을 파악한다.

---

## Step 2: 컨텍스트 수집

작업에 필요한 코드베이스 컨텍스트를 수집한다. 이 단계는 에이전트 투입 전에 수행하여, `{N}`명의 에이전트에게 동일한 기반 정보를 제공한다.

수집 항목:
1. **프로젝트 구조**: `ls`, `Glob`으로 디렉토리/파일 구조 파악
2. **CLAUDE.md**: 프로젝트의 코딩 컨벤션, 아키텍처 패턴 확인
3. **관련 파일 탐색**: `Grep`, `Glob`으로 이슈와 관련된 파일 식별
4. **참조 체인 추적**: `Grep`과 `LSP`(사용 가능한 경우)를 병행하여 코드 간 참조 관계를 꼼꼼하게 파악
5. **최근 변경 이력**: `git log`로 관련 영역의 최근 커밋 확인

LSP가 사용 가능한 환경에서는 `Go to Definition`, `Find References` 등을 활용하여 Grep만으로는 놓칠 수 있는 암묵적 참조(인터페이스 구현, 상속 체인 등)도 추적한다.

---

## Step 3: 병렬 계획 수립

`{N}`개의 에이전트를 **동시에** 실행한다. 각 에이전트는 동일한 컨텍스트를 받고, 독립적으로 구현 계획서를 작성한다.

### 모델별 에이전트 실행 방식

#### Claude 모델 (opus / sonnet / haiku)

Agent 도구를 사용한다. `{N}`개의 Agent 호출을 **하나의 메시지에서 병렬로** 실행한다.

```
Agent({
  description: "Plan Storm 에이전트 {i}/{N}",
  subagent_type: "general-purpose",
  model: "{MODEL}",
  prompt: "{아래 에이전트 프롬프트 템플릿}"
})
```

- `model` 파라미터에 `{MODEL}` 값(opus/sonnet/haiku)을 그대로 전달한다.
- `{N}`개의 Agent 호출을 단일 메시지에 모두 포함하여 병렬 실행한다.

#### LM Studio 모델 (lmstudio)

Agent 도구는 LM Studio를 직접 지원하지 않으므로, Bash 도구로 OpenAI 호환 API를 호출한다.

`{N}`개의 Bash 호출을 **하나의 메시지에서 병렬로** 실행한다:

```bash
curl -s --max-time 300 http://localhost:1234/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "{LMSTUDIO_MODEL}",
    "messages": [
      {"role": "system", "content": "당신은 구현 계획을 수립하는 시니어 엔지니어입니다."},
      {"role": "user", "content": "{아래 에이전트 프롬프트 템플릿의 내용}"}
    ],
    "temperature": 0.7,
    "max_tokens": 4096
  }'
```

- **타임아웃**: localhost 호출은 최대 **5분(300초)**으로 설정한다 (`--max-time 300`).
- Bash 도구의 `timeout` 파라미터도 **300000ms**(5분)로 설정한다.
- JSON 내부의 특수문자(줄바꿈, 따옴표 등)는 반드시 이스케이프 처리한다.
- 각 응답의 `.choices[0].message.content`를 계획서로 사용한다.
- 응답 실패 시 해당 에이전트 번호를 사용자에게 알리고, 나머지 성공한 계획서로 계속 진행할지 물어본다. 성공한 계획서가 2개 미만이면 스킬을 중단한다.

### 에이전트 프롬프트 템플릿

모델 종류에 관계없이 모든 에이전트에게 동일한 프롬프트를 전달한다:

```
당신은 구현 계획을 수립하는 시니어 엔지니어입니다.

## 작업 요청
{이슈 내용 또는 작업 설명}

## 프로젝트 컨텍스트
{Step 2에서 수집한 컨텍스트: 프로젝트 구조, 관련 파일 목록, CLAUDE.md 내용 등}

## 분석 관점

다음 3가지 관점을 모두 포함하여 계획서를 작성하세요:

### 1. Architecture (구조/설계)
- 프로젝트의 기존 아키텍처와 디렉토리 구조를 분석하고, 그에 맞는 설계 방향을 제시
- 모듈 간 의존성과 책임 분리가 올바른지 검토
- 새로운 파일/클래스를 추가할 경우 기존 패턴과의 일관성 확인
- 프로젝트의 CLAUDE.md나 컨벤션 문서가 있다면 반드시 준수

### 2. Implementation (구현)
- 수정/생성해야 할 파일의 전체 목록 (경로 포함)
- 각 파일별 구체적인 변경 내용
- 참조 체인: 변경이 영향을 미치는 다른 파일들 (Grep과 LSP로 확인)
- 구현 순서 (의존성 기반 — 어떤 파일을 먼저 수정해야 하는지)
- 빌드 영향도: 빌드 깨짐 가능성이 있는 변경 식별

### 3. Review (품질/누락 검토)
- 누락된 파일이나 작업이 없는지 확인
- 엣지 케이스 식별
- 필요한 마이그레이션, 테스트, 설정 변경 등 부수 작업 확인
- 보안 고려사항
- 성능 영향 검토

## 출력 형식

아래 형식으로 계획서를 작성하세요:

### 작업 요약
(1-3문장으로 작업 내용 요약)

### 설계 방향
(아키텍처 관점의 핵심 결정사항)

### 변경 파일 목록
| 파일 경로 | 작업 유형 | 변경 내용 |
|-----------|----------|----------|
| ... | 생성/수정/삭제 | ... |

### 구현 순서
1. (의존성 기반으로 정렬된 작업 순서)
2. ...

### 참조 영향도
(변경으로 인해 영향받는 다른 파일/모듈)

### 리스크 및 주의사항
- ...

### 누락 검토 체크리스트
- [ ] ...
```

`{N}`개 에이전트 모두 같은 프롬프트를 받되, 각자 독립적으로 분석하므로 서로 다른 관점과 발견이 나온다.

## Step 4: 교차 리뷰

`{N}`개의 계획서가 모두 완료되면, **하나의 리뷰 에이전트**를 실행하여 교차 리뷰를 수행한다.

리뷰 에이전트는 `{MODEL}`과 동일한 모델을 사용한다 (Claude 모델이면 Agent 도구, lmstudio이면 Bash+curl). lmstudio 사용 시 타임아웃은 동일하게 **5분(300초)**을 적용한다.

### 리뷰 에이전트 프롬프트

계획서 섹션은 에이전트 수에 따라 동적으로 구성한다. 각 계획서에 알파벳 라벨(A, B, C, ...)을 부여한다.

```
당신은 시니어 테크 리드로서, {N}명의 엔지니어가 독립적으로 작성한 구현 계획서를 교차 리뷰합니다.

## 원본 작업 요청
{이슈 내용}

## 계획서 A
{에이전트 1의 계획서}

## 계획서 B
{에이전트 2의 계획서}

{... N개까지 반복, 알파벳 라벨 순서대로 ...}

## 리뷰 지침

무조건적으로 비판하지 말고, 각 계획서가 올바른 방향인지 건설적으로 검토하세요:

1. **합의 사항**: {N}개 계획서가 공통으로 동의하는 부분 (높은 신뢰도)
2. **차이점 분석**: 계획서 간 불일치가 있는 부분과 그 이유
3. **누락 탐지**: 특정 계획서에만 있고 다른 계획서에는 빠진 중요한 항목
4. **방향성 검증**: 작업의 전체 방향이 요구사항에 부합하는지
5. **참조 체인 검증**: 변경 영향을 받는 파일이 모든 계획서에서 충분히 커버되었는지
6. **실행 가능성**: 제안된 구현 순서가 현실적인지, 빌드가 단계별로 깨지지 않는지

## 출력 형식

### 합의 사항 (높은 신뢰도)
- ...

### 핵심 차이점 및 권장 방향
| 항목 | {각 계획서별 열: A, B, ... N번째까지} | 권장 |
|------|{동적으로 계획서 수만큼 열 생성}|------|
| ... | ... | ... |

### 누락된 항목 (특정 계획서에서만 발견)
- ...

### 리스크 및 주의사항 (통합)
- ...

### 최종 권장 구현 순서
1. ...
```

## Step 5: 최종 통합 및 저장

교차 리뷰 결과를 기반으로 최종 계획서를 작성한다.

### 저장 위치

```
.claude/tasks/{title}.md
```

- `{title}`: 작업 내용을 요약한 kebab-case 이름 (예: `remove-tenant-mapping`, `add-search-endpoint`, `refactor-auth-middleware`)
- 디렉토리가 없으면 자동 생성한다

### 최종 계획서 형식

```markdown
# 작업 계획서: {작업 제목}

> 생성일: {날짜}
> 브랜치: {branch-name}
> 소스: {GitHub 이슈 URL 또는 "수동 입력"}
> 에이전트: {N}명
> 모델: {MODEL} {lmstudio인 경우: "({LMSTUDIO_MODEL})"}

## 작업 요약
(요구사항 요약)

## 설계 방향
(아키텍처/구조 관점의 핵심 결정사항)

## 변경 파일 목록
| # | 파일 경로 | 작업 유형 | 변경 내용 |
|---|-----------|----------|----------|
| 1 | ... | 생성/수정/삭제 | ... |

## 구현 순서
(의존성 기반, 빌드가 단계별로 깨지지 않는 순서)

1. [ ] 작업 1
2. [ ] 작업 2
...

## 참조 영향도
(변경으로 인해 영향받는 파일/모듈 목록)

## 리스크 및 주의사항
- ...

## 에이전트 리뷰 요약
### 합의 사항
- ...

### 핵심 논의 사항
- ...
```

### 완료 후

최종 계획서 저장 후 사용자에게 결과를 요약하여 보여준다:
- 계획서 저장 경로
- 사용된 에이전트 수 및 모델
- 주요 변경 파일 수
- 핵심 설계 결정사항 (1-2줄)
- 특별히 주의할 리스크가 있다면 언급
